## Model Diagnosis (모델 진단)

모델 진단은 모델 구조에 대한 종합적인 분석을 수행하고, 실행 가능한 최적화 권장 사항을 생성합니다.

### 아키텍처 감지

ssook은 그래프 내 연산 집합을 알려진 패턴과 매칭하여 모델 아키텍처를 자동으로 식별합니다:

| 아키텍처 | 주요 연산 |
|----------|-----------|
| **YOLO** | Conv, Concat, Resize, Sigmoid |
| **RF-DETR** | MultiHeadAttention, LayerNormalization, Conv, Gather |
| **EVA-02** | LayerNormalization, Attention, Gelu, MatMul |
| **Transformer** | LayerNormalization, Attention, MatMul, Softmax |
| **CNN** | Conv 존재, Attention 연산 없음 |
| **Hybrid** | Conv와 Attention 연산 모두 존재 |

패턴 연산의 50% 이상이 존재하면 해당 아키텍처로 식별하는 오버랩 스코어링 방식을 사용합니다. 이를 통해 모델 유형에 맞는 최적화 권장 사항을 제공합니다.

### 가중치 분석

Conv, MatMul, Gemm, ConvTranspose 레이어의 각 가중치 텐서에 대해 다음을 계산합니다:

- **범위** (최솟값/최댓값): 값이 얼마나 넓게 분포하는지
- **평균 / 표준편차**: 분포의 중심과 퍼짐 정도
- **희소성**: 0에 가까운 가중치의 비율 (|w| < 10⁻⁷) — 기존 가지치기 적용 여부나 자연적으로 희소한 레이어를 나타냄
- **이상치 비율**: 3 표준편차를 넘는 가중치의 비율 — 이상치 비율이 높으면 양자화가 어려워짐

### 양자화 적합성

모델이 양자화에 얼마나 잘 반응할지 평가합니다:

- **양자화 가능 비율**: 양자화할 수 있는 연산의 비율. Conv, MatMul, Gemm, Relu, MaxPool 등은 양자화 가능합니다. 커스텀 연산이나 특정 정규화 레이어는 불가능합니다.
- **노드별 민감도**: 양자화 가능한 각 레이어에 민감도 점수가 부여됩니다:
  ```
  sensitivity = max(|weights|) × (1 + outlier_ratio × 10)
  ```
  민감도가 높은 레이어(넓은 가중치 범위 + 많은 이상치)는 양자화 시 정확도 손실이 더 큽니다.
- **양자화 불가 연산**: 양자화할 수 없는 연산 목록을 제공하여, 전체 INT8과 Mixed Precision 중 어떤 것을 선택할지 판단하는 데 도움을 줍니다.

### 가지치기 가능성

각 Conv 레이어에 대해 다음을 분석합니다:

- **채널 중요도**: 각 출력 채널 가중치의 L1-norm. L1-norm이 매우 낮은 채널은 출력에 기여하는 바가 적습니다.
- **저중요도 채널**: L1-norm이 레이어 평균의 10% 미만인 채널 수 — Channel Pruning 후보입니다.
- **전체 희소성**: 모든 레이어에 걸친 0에 가까운 가중치의 비율 — Weight Pruning이 이미 적용되었거나 적용 가능한 정도를 나타냅니다.

### 그래프 효율성

- **메모리 바운드 연산 비율**: 연산 바운드(compute-bound)가 아닌 메모리 바운드(memory-bound) 연산(Reshape, Transpose, Concat, Gather, Slice 등)의 비율. 비율이 높으면 모델이 연산보다 데이터 이동에 더 많은 시간을 소비한다는 의미입니다.
- **융합 가능 패턴**: 단일 커널로 융합할 수 있는 인접 연산 쌍의 수 (Conv+BN, Conv+Relu, MatMul+Add, Conv+Add+Relu). 융합 가능 패턴이 많을수록 그래프 최적화의 여지가 큽니다.

### Health Score (건강 점수)

모델의 최적화 상태를 요약하는 0~100점 단일 점수:

```
health_score = 100 - (critical_findings × 20) - (warning_findings × 5)
```

- **100**: 모델이 잘 최적화된 상태
- **60~80**: 일부 최적화 기회가 존재
- **60 미만**: 상당한 최적화가 권장됨

### 추천 엔진

진단 결과를 바탕으로 ssook은 우선순위별 최적화 권장 목록을 생성합니다:

| 우선순위 | 방법 | 권장 조건 |
|----------|------|-----------|
| 0 | Dead Node Elimination | 항상 — 먼저 그래프를 정리 |
| 1 | ORT Graph Optimizer | 융합 가능 패턴이 감지될 때 |
| 2 | Dynamic INT8 | 양자화 가능 비율 > 50% |
| 2 | FP16 | 모델 크기 > 10 MB |
| 2 | Weight Pruning | 희소성 < 50% 이고 파라미터 > 100K |
| 2 | Channel Pruning | 저중요도 채널 > 5% |
| 3 | Static INT8 / Mixed Precision | 양자화 가능 비율 > 50% (양자화 불가 연산 유무에 따라) |

각 권장 사항에는 다음이 포함됩니다:
- **이유**: 이 최적화가 제안되는 이유
- **예상 효과**: 예상 속도 향상 또는 크기 감소
- **실행 가능 여부**: ssook에서 직접 적용 가능한지 (vs. PyTorch 등 외부 도구 필요)

**사용 방법:**
1. **Tools** → **Model Optimization**으로 이동
2. 모델 로드 후 **Diagnose** 클릭
3. Health Score, 발견 사항, 권장 사항 검토
4. 권장 사항에서 **Apply**를 클릭하여 실행하거나, 커스텀 파이프라인 구성

## Model Profiler (모델 프로파일러)

프로파일러는 실제 런타임 성능을 측정하고 연산 비용을 추정합니다.

### 레이어별 타이밍

**동작 원리:**
1. `SessionOptions.enable_profiling`을 통해 ONNX Runtime의 내장 프로파일링을 활성화합니다.
2. 모델의 예상 입력 형태에 맞는 더미 입력으로 추론을 실행합니다.
3. ORT가 노드별 타이밍 이벤트가 포함된 JSON 트레이스 파일을 작성합니다.
4. ssook이 카테고리 `"Node"`인 이벤트를 파싱하고 각 연산의 `dur` (마이크로초 단위 지속시간)을 추출합니다.

결과는 실행 시간 기준으로 정렬된 레이어 목록으로, 모델이 어디에서 시간을 소비하는지 정확히 보여줍니다.

### FLOPs & MACs 추정

ssook은 각 레이어의 연산 비용을 추정합니다:

| 연산 | FLOPs 공식 |
|------|-----------|
| **Conv** | 2 × C_out × C_in × K_h × K_w × H_out × W_out |
| **ConvTranspose** | 2 × C_in × C_out × K_h × K_w × H_out × W_out |
| **MatMul / Gemm** | 2 × M × K × N |
| **BatchNormalization** | 4 × elements (평균, 분산, 정규화, 스케일+시프트) |

MACs (Multiply-Accumulate 연산) = FLOPs / 2. 이 추정치는 형태 추론을 사용하여 출력 차원을 결정합니다.

### 메모리 분석

- **가중치 메모리**: 전체 파라미터 수 × 4바이트 (FP32). MB 단위로 표시.
- **최대 활성화 메모리**: 가장 큰 중간 텐서 × 3으로 추정 (메모리에 동시에 존재하는 텐서에 대한 대략적 추정).
- **전체 메모리**: 가중치 메모리 + 최대 활성화 메모리.

### 병목 진단

100μs 이상 소요되는 레이어를 다음과 같이 분류합니다:

| 카테고리 | 연산 | 의미 |
|----------|------|------|
| **Compute** | Conv, MatMul, Gemm, ConvTranspose | CPU/GPU 바운드 — 양자화나 가지치기로 개선 가능 |
| **Memory** | Reshape, Transpose, Concat, Gather, Slice | 메모리 바운드 — 그래프 최적화나 융합으로 개선 가능 |
| **Other** | 기타 모든 연산 | 특수 최적화가 도움될 수 있음 |

심각도 수준: **High** (>1000μs), **Medium** (>500μs), **Low** (>100μs).

프로파일러는 다음도 보고합니다:
- **양자화 가능 비율**: 양자화할 수 있는 연산의 비율
- **예상 INT8 속도 향상**: 양자화 가능 비율에 기반한 대략적 추정 (완전 양자화 가능 모델의 경우 최대 ~3배)
- **그래프 깊이**: 출력에서 입력까지의 최장 경로 (깊은 그래프는 순차적 병목이 더 많을 수 있음)

**사용 방법:**
1. **Tools** → **Model Profiler**로 이동
2. ONNX 모델 로드
3. 실행 횟수 (기본값: 20) 및 워밍업 반복 횟수 (기본값: 3) 설정
4. **Profile** 클릭
5. 지연시간 통계 (avg, P50, P95, P99), 주요 병목 레이어, FLOPs 분석, 최적화 제안 검토

## Model Inspector (모델 인스펙터)

인스펙터는 모델의 구조와 호환성에 대한 빠른 개요를 제공합니다.

### 그래프 메타데이터

- **파일 크기** (MB)
- **Opset 버전**: ONNX 연산자 집합 버전. 높은 버전일수록 더 많은 연산을 지원합니다. 양자화에는 opset ≥ 11이 필요합니다.
- **IR 버전**: ONNX 중간 표현 버전.
- **Producer**: 모델을 내보낸 프레임워크 (예: "pytorch", "tf2onnx").
- **노드 수**: 그래프 내 전체 연산 수.
- **연산 수 분석**: 연산 유형별 분석 (예: Conv: 52, Relu: 48, BatchNormalization: 24).
- **파라미터 수**: 모든 이니셜라이저에 걸친 전체 학습 가능 파라미터 수.

### 입출력 형태

각 입력 및 출력 텐서에 대해:
- **이름**: 그래프에서 사용되는 텐서 이름
- **형태**: 차원 (동적 차원인 `"batch"` 또는 `"height"` 포함 가능)
- **데이터 타입**: 예: `tensor(float)`, `tensor(float16)`, `tensor(int64)`

모델이 어떤 입력을 기대하고 어떤 출력을 생성하는지 이해하는 데 유용합니다.

### EP 호환성

ssook은 어떤 Execution Provider가 모델을 로드할 수 있는지 테스트하고, 각 EP가 모델의 연산을 얼마나 잘 지원하는지 추정합니다:

- **사용 가능한 EP**: 시스템에 설치된 모든 EP
- **호환 가능한 EP**: 모델을 성공적으로 로드할 수 있는 EP
- **EP별 분석**: 호환 가능한 각 EP에 대해 ssook이 추정하는 항목:
  - **지원 연산**: EP에서 네이티브로 실행되는 연산
  - **폴백 연산**: CPU로 폴백되는 연산
  - **지원 비율**: EP에서 실행되는 연산의 비율 (높을수록 GPU 활용도가 좋음)

이는 실제 GPU 실행 없이도 호환성을 추정하기 위해 알려진 EP 연산 지원(CUDA, TensorRT, OpenVINO, DirectML)의 휴리스틱 매핑을 사용합니다.

**사용 방법:**
1. **Tools** → **Model Inspector**로 이동
2. ONNX 모델 로드
3. 그래프 메타데이터, 입출력 형태, EP 호환성 보고서 검토
