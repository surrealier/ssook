## Quantization (양자화)

양자화는 모델 가중치(및 선택적으로 활성화 값)를 고정밀 부동소수점(FP32)에서 저정밀 형식(INT8, FP16)으로 변환합니다. 이를 통해 모델 크기가 줄어들고, 특히 CPU에서 추론 속도가 빨라집니다.

### Dynamic INT8

**동작 원리:**
- 가중치는 **내보내기 시점**에 FP32에서 INT8로 변환됩니다.
- 활성화 값은 FP32로 유지되며, 추론 중에 **실시간으로** 양자화됩니다.
- 보정 데이터가 필요 없습니다 — 가중치의 양자화 범위는 가중치 값에서 직접 계산됩니다.

**장단점:**
- ✅ 가장 간단한 방법 — 모델을 선택하고 실행하기만 하면 됩니다
- ✅ 모델 크기 약 50% 감소
- ✅ CPU에서 1.5~2배 속도 향상
- ⚠️ 활성화 범위를 런타임에 추정하므로 Static 양자화보다 최적화 수준이 낮습니다

**ssook에서 사용하는 방법:**
1. **Tools** 탭 → **Calibration / Quantization**으로 이동
2. ONNX 모델 로드
3. **Dynamic INT8** 선택
4. 가중치 타입 선택 (UInt8 또는 Int8)
5. **Run** 클릭

### Static INT8

**동작 원리:**
- 가중치와 활성화 값 모두 INT8로 변환됩니다.
- **보정 데이터셋** (대표 이미지 50~200장)을 사용하여 추론 중 실제 활성화 범위를 측정합니다.
- ssook이 보정 이미지를 모델의 입력 형태에 맞게 자동으로 전처리합니다 (리사이즈, 정규화, 배치).
- QDQ (Quantize-Dequantize) 형식은 양자화 가능한 연산 주위에 Q/DQ 노드를 삽입하여, 최대 호환성을 위해 그래프 구조를 유지합니다.

**장단점:**
- ✅ 최고 속도 향상: CPU에서 2~4배
- ✅ 모델 크기 약 75% 감소
- ✅ 보정 데이터가 대표성이 있으면 정확도 손실 최소화
- ⚠️ 대상 도메인의 보정 이미지가 필요합니다

**ssook에서 사용하는 방법:**
1. **Tools** → **Calibration / Quantization**으로 이동
2. ONNX 모델 로드
3. **Static INT8** 선택
4. 보정 이미지 디렉토리와 최대 이미지 수 설정
5. 옵션 구성 (per-channel, QDQ 형식)
6. **Run** 클릭 — 진행률 표시줄에 보정 진행 상황이 표시됩니다

### FP16 변환

**동작 원리:**
- 모든 FP32 가중치와 상수가 FP16 (반정밀도 부동소수점)으로 변환됩니다.
- FP16은 32비트 대신 16비트를 사용하여 값당 저장 공간이 절반으로 줄어듭니다.
- 표현 가능 범위가 더 작지만 (±65,504 vs ±3.4×10³⁸), 대부분의 신경망 가중치에는 충분합니다.

**장단점:**
- ✅ 모델 크기 약 50% 감소
- ✅ 대부분의 모델에서 정확도 손실이 거의 없음
- ✅ FP16 하드웨어가 있는 GPU에서 상당한 속도 향상 (NVIDIA Tensor Cores, Apple Neural Engine)
- ⚠️ CPU에서는 속도 향상이 미미함 (CPU의 FP16 지원은 제한적)
- ⚠️ `onnxconverter-common` 패키지가 필요합니다

**ssook에서 사용하는 방법:**
1. **Tools** → **Calibration / Quantization**으로 이동
2. ONNX 모델 로드
3. **FP16** 선택
4. **Run** 클릭

### Mixed Precision (혼합 정밀도)

**동작 원리:**
- 모든 레이어가 양자화에 동일하게 반응하지는 않습니다. 가중치 범위가 넓거나 이상치가 많은 레이어는 양자화 시 정확도 손실이 더 큽니다.
- ssook은 양자화 가능한 각 레이어에 대해 **민감도 점수**를 계산합니다:
  ```
  sensitivity = weight_range × (1 + outlier_ratio × 10)
  ```
  여기서 `outlier_ratio`는 3 표준편차를 넘는 가중치의 비율입니다.
- 가장 민감한 상위 N%의 레이어는 INT8 양자화에서 **제외**되어 FP32로 유지됩니다.
- 나머지 레이어는 정상적으로 INT8로 양자화됩니다.

**장단점:**
- ✅ 전체 INT8보다 더 나은 정확도 — 민감한 레이어가 FP32로 유지됨
- ✅ 여전히 2~3배 속도 향상 달성
- ✅ 자동 처리 — 수동 레이어 선택이 필요 없음
- ⚠️ 전체 INT8보다 압축률이 약간 낮음

**ssook에서 사용하는 방법:**
1. **Tools** → **Calibration / Quantization**으로 이동
2. ONNX 모델 로드
3. **Mixed Precision** 선택
4. 제외 비율 설정 (기본값: 가장 민감한 레이어 20%)
5. **Run** 클릭

## Pruning (가지치기)

가지치기는 모델에서 불필요한 가중치나 채널을 제거하여 크기와 연산 비용을 줄입니다.

### Weight Pruning (가중치 가지치기)

**동작 원리:**
- **크기 기반 비구조적 가지치기**: 절대값이 가장 작은 가중치는 출력에 기여하는 바가 적으므로 0으로 설정할 수 있습니다.
- `sparsity_ratio` (예: 0.3)가 주어지면, 크기 기준 하위 30%의 가중치가 0으로 처리됩니다.
- Conv, MatMul, Gemm, ConvTranspose 연산의 가중치 텐서를 대상으로 합니다.
- 결과적으로 생성된 희소 모델은 압축 시 더 작아지며, 희소 연산을 지원하는 런타임에서 더 빠를 수 있습니다.

**장단점:**
- ✅ 10~30% 크기 감소 (압축 시 더 큰 효과)
- ✅ 적당한 희소성(≤30%)에서 정확도 손실 최소화
- ⚠️ 속도 향상은 런타임의 희소 연산 지원에 따라 다름 — 표준 ONNX Runtime은 희소 연산을 가속하지 않음
- ⚠️ 높은 희소성(>50%)은 정확도를 눈에 띄게 저하시킬 수 있음

**ssook에서 사용하는 방법:**
1. **Tools** → **Model Optimization**으로 이동
2. ONNX 모델 로드
3. **Weight Pruning** 선택
4. 희소성 비율 설정 (0.0~1.0, 권장: 0.2~0.3)
5. **Run** 클릭

### Channel Pruning (채널 가지치기)

**동작 원리:**
- **L1-norm 기반 구조적 가지치기**: 각 Conv 레이어에서 출력 채널별 가중치의 L1-norm (절대값의 합)을 계산합니다.
- L1-norm이 가장 낮은 채널은 해당 레이어의 출력에 기여하는 바가 적습니다.
- L1-norm 기준 하위 N%의 채널이 완전히 제거됩니다 — 가중치와 해당 바이어스 값 모두 제거됩니다.
- Weight Pruning과 달리 이것은 **구조적** 가지치기입니다: 결과 모델은 채널 수가 줄어들어 특별한 희소 런타임 없이도 모든 하드웨어에서 더 빠르게 실행됩니다.

**장단점:**
- ✅ 가지치기된 채널에 비례하여 FLOPs 직접 감소
- ✅ 모든 하드웨어에서 더 빠름 (희소 런타임 불필요)
- ⚠️ 정확도 회복을 위해 파인튜닝이 필요할 수 있음
- ⚠️ 레이어가 0개 채널로 가지치기되는 것을 방지하기 위해 최소 채널 수가 적용됨

**ssook에서 사용하는 방법:**
1. **Tools** → **Model Optimization**으로 이동
2. ONNX 모델 로드
3. **Channel Pruning** 선택
4. 가지치기 비율 (권장: 0.1~0.3) 및 최소 채널 수 (기본값: 4) 설정
5. **Run** 클릭

## Graph Optimization (그래프 최적화)

그래프 최적화는 모델의 수학적 동작을 변경하지 않으면서 연산 그래프를 재구성하여 중복을 제거하고 연산을 융합합니다.

### ORT Graph Optimizer

**동작 원리:**
- ONNX Runtime의 내장 그래프 최적화 패스를 사용합니다.
- **Constant Folding (상수 접기)**: 입력이 모두 상수인 연산을 미리 계산합니다 (예: Shape → Gather → Unsqueeze 체인).
- **Operator Fusion (연산자 융합)**: 인접한 연산을 하나의 최적화된 커널로 병합합니다:
  - Conv + BatchNormalization → 단일 Conv (BN 파라미터가 Conv 가중치에 통합)
  - Conv + Relu → ConvRelu 융합 커널
  - MatMul + Add → 융합된 Gemm
  - Conv + Add + Relu → 단일 융합 커널
- 세 가지 최적화 수준: **Basic** (상수 접기), **Extended** (+ 공통 부분식 제거), **All** (+ 고급 융합).

**장단점:**
- ✅ 5~15% 지연시간 감소
- ✅ 정확도 손실 없음 — 수학적으로 동일
- ✅ 항상 안전하게 적용 가능

### ONNX Simplifier

**동작 원리:**
- `onnxsim` 라이브러리를 사용하여 ONNX 그래프 수준에서 형태 추론과 상수 접기를 수행합니다.
- 가능한 경우 동적 형태를 해결하고, 중복 연산을 단순화하여 더 깔끔한 그래프를 생성합니다.

**장단점:**
- ✅ 더 깔끔한 그래프, 종종 더 작은 파일
- ✅ 정확도 손실 없음
- ⚠️ `onnxsim` 패키지가 필요합니다

### Dead Node Elimination (불필요 노드 제거)

**동작 원리:**
- 출력 노드에서 역방향으로 그래프를 추적하여 도달 가능한 모든 노드를 찾습니다.
- 출력에서 도달할 수 없는 노드는 모두 제거됩니다.
- 모델 내보내기나 이전 변환에서 남은 불필요한 노드를 정리합니다.

**장단점:**
- ✅ 그래프 크기 감소
- ✅ 정확도 손실 없음
- ✅ 다른 최적화 전에 실행하면 좋음

**ssook에서 그래프 최적화 사용하는 방법:**
1. **Tools** → **Model Optimization**으로 이동
2. ONNX 모델 로드
3. 최적화 방법 선택
4. ORT Graph Optimizer의 경우 수준 선택 (Basic / Extended / All)
5. **Run** 클릭

## 최적화 파이프라인

ssook에서는 여러 최적화를 **순차적으로 연결**할 수 있습니다. 예를 들어:

1. Dead Node Elimination → 그래프 정리
2. ORT Graph Optimizer → 연산 융합
3. Channel Pruning → 중요하지 않은 채널 제거
4. Dynamic INT8 Quantization → 가중치 압축

각 단계는 이전 단계의 출력을 입력으로 사용합니다. 파이프라인은 각 단계에서의 크기 변화와 결과를 추적합니다.

**사용 방법:**
1. **Tools** → **Model Optimization**으로 이동
2. **Model Diagnosis** 기능을 사용하여 권장 최적화 단계 확인
3. 진단 엔진이 모델 특성에 기반한 우선순위별 파이프라인을 제안합니다
4. 권장 파이프라인을 적용하거나 단계를 직접 커스터마이즈

## 성능 비교

CPU (Intel i7-1165G7) 기준 일반적인 결과:

| 모델 | 방법 | 크기 | 추론 시간 | mAP@50 |
|------|------|------|-----------|--------|
| YOLOv8n FP32 | 원본 | 12.2 MB | ~45 ms | 37.3 |
| YOLOv8n INT8 (Dynamic) | Dynamic Quantization | 3.4 MB | ~25 ms | 36.8 |
| YOLOv8n INT8 (Static) | Static Quantization | 3.2 MB | ~18 ms | 37.0 |
| YOLOv8n FP16 | FP16 변환 | 6.1 MB | ~38 ms | 37.3 |

> 결과는 하드웨어와 모델에 따라 다릅니다. 최적화 후에는 반드시 ssook의 Evaluation 탭에서 정확도를 검증하세요.

## 팁 & 주의사항

1. **Graph Optimization부터 시작하세요** — 정확도 손실이 없으며, 이후 최적화의 효과를 높여줍니다.
2. **먼저 Model Diagnosis를 사용하세요** — 모델을 분석하고 최적의 최적화 전략을 추천합니다.
3. **양자화 후 정확도를 검증하세요** — Evaluation 탭에서 최적화 전후의 mAP를 비교하세요.
4. **보정 데이터가 중요합니다** — Static INT8의 경우, 실제 배포 도메인의 이미지를 사용하세요 (보통 50~200장이면 충분합니다).
5. **opset 버전을 확인하세요** — 양자화에는 opset ≥ 11이 필요합니다. 낮은 opset의 모델은 실패할 수 있습니다.
6. **CenterNet / DETR 모델** — 일부 커스텀 연산은 INT8을 지원하지 않을 수 있습니다. FP16 또는 Mixed Precision을 대신 사용하세요.
7. **Channel Pruning은 파인튜닝이 필요합니다** — 최상의 결과를 위해 가지치기 후 학습 데이터로 모델을 파인튜닝하세요.
