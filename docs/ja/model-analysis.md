## Model Diagnosis（モデル診断）

Model Diagnosisは、モデルの構造を包括的に分析し、実行可能な最適化の推奨事項を生成します。

### アーキテクチャ検出

ssookは、グラフ内の演算セットを既知のパターンと照合することで、モデルアーキテクチャを自動的に識別します：

| アーキテクチャ | 主要な演算 |
|-------------|---------------|
| **YOLO** | Conv, Concat, Resize, Sigmoid |
| **RF-DETR** | MultiHeadAttention, LayerNormalization, Conv, Gather |
| **EVA-02** | LayerNormalization, Attention, Gelu, MatMul |
| **Transformer** | LayerNormalization, Attention, MatMul, Softmax |
| **CNN** | Convあり、Attention演算なし |
| **Hybrid** | ConvとAttention演算の両方あり |

検出にはオーバーラップスコアリングを使用します — パターンの演算の50%以上が存在する場合、そのアーキテクチャが識別されます。これにより、モデルタイプに合わせた最適化の推奨が可能になります。

### 重み分析

Conv、MatMul、Gemm、ConvTransposeレイヤーの各重みテンソルについて、診断は以下を計算します：

- **範囲**（min/max）：値の広がり
- **平均 / 標準偏差**：分布の中心と広がり
- **スパース性**：ゼロに近い重みの割合（|w| < 10⁻⁷）— 既存の枝刈りまたは自然にスパースなレイヤーを示す
- **外れ値比率**：3標準偏差を超える重みの割合 — 外れ値比率が高いと量子化が困難になる

### 量子化適合性

診断は、モデルが量子化にどの程度適しているかを評価します：

- **量子化可能比率**：量子化できる演算の割合。Conv、MatMul、Gemm、Relu、MaxPoolなどは量子化可能です。カスタム演算や特定の正規化レイヤーは量子化できません。
- **ノードごとの感度**：各量子化可能レイヤーに感度スコアが付与されます：
  ```
  sensitivity = max(|weights|) × (1 + outlier_ratio × 10)
  ```
  感度が高いレイヤー（重みの範囲が広く外れ値が多い）は、量子化時により多くの精度を失います。
- **量子化不可能な演算**：量子化できない演算をリストアップし、完全INT8とMixed Precisionのどちらを選ぶかの判断を支援します。

### 枝刈りポテンシャル

各Convレイヤーについて、診断は以下を分析します：

- **チャネル重要度**：各出力チャネルの重みのL1ノルム。L1ノルムが非常に低いチャネルは出力への寄与が少ない。
- **低重要度チャネル**：L1ノルムがレイヤー平均の10%未満のチャネル数 — Channel Pruningの候補。
- **全体的なスパース性**：全レイヤーにわたるゼロに近い重みの割合 — Weight Pruningがどの程度適用済みか、または適用可能かを示す。

### グラフ効率性

- **メモリバウンド演算比率**：計算バウンドではなくメモリバウンドの演算（Reshape、Transpose、Concat、Gather、Sliceなど）の割合。比率が高い場合、モデルは計算よりもデータ移動に多くの時間を費やしています。
- **融合可能パターン**：単一カーネルに融合できる隣接演算ペアの数（Conv+BN、Conv+Relu、MatMul+Add、Conv+Add+Relu）。融合可能パターンが多いほど、グラフ最適化の余地が大きい。

### Health Score（健全性スコア）

モデルの最適化状態を要約する0〜100の単一スコア：

```
health_score = 100 - (critical_findings × 20) - (warning_findings × 5)
```

- **100**：モデルは十分に最適化されている
- **60〜80**：いくつかの最適化の余地がある
- **60未満**：大幅な最適化を推奨

### 推奨エンジン

診断に基づき、ssookは優先順位付きの最適化推奨リストを生成します：

| 優先度 | 手法 | 推奨条件 |
|----------|--------|-----------------|
| 0 | Dead Node Elimination | 常に — まずグラフをクリーンアップ |
| 1 | ORT Graph Optimizer | 融合可能パターンが検出された場合 |
| 2 | Dynamic INT8 | 量子化可能比率 > 50%の場合 |
| 2 | FP16 | モデル > 10 MBの場合 |
| 2 | Weight Pruning | スパース性 < 50%かつパラメータ > 100Kの場合 |
| 2 | Channel Pruning | 低重要度チャネル > 5%の場合 |
| 3 | Static INT8 / Mixed Precision | 量子化可能比率 > 50%の場合（量子化不可能な演算の有無による） |

各推奨には以下が含まれます：
- **理由**：この最適化が提案される理由
- **期待される効果**：推定される高速化またはサイズ削減
- **実行可能性**：ssookで直接適用できるか（外部ツール（PyTorchなど）が必要か）

**使い方：**
1. **Tools** → **Model Optimization** に移動
2. モデルを読み込んだ後、**Diagnose** をクリック
3. Health Score、所見、推奨事項を確認
4. 任意の推奨事項の **Apply** をクリックして実行するか、カスタムパイプラインを構築

## Model Profiler（モデルプロファイラ）

プロファイラは、実際のランタイム性能を測定し、計算コストを推定します。

### レイヤーレベルタイミング

**仕組み：**
1. `SessionOptions.enable_profiling` を通じてONNX Runtimeの組み込みプロファイリングを有効化。
2. モデルの期待する形状に一致するダミー入力で推論を実行。
3. ORTがノードごとのタイミングイベントを含むJSONトレースファイルを出力。
4. ssookがカテゴリ `"Node"` のイベントを解析し、各演算の `dur`（マイクロ秒単位の所要時間）を抽出。

結果は実行時間順にランク付けされたレイヤーのリストで、モデルがどこに時間を費やしているかを正確に示します。

### FLOPs＆MACs推定

ssookは各レイヤーの計算コストを推定します：

| 演算 | FLOPs計算式 |
|-----------|--------------|
| **Conv** | 2 × C_out × C_in × K_h × K_w × H_out × W_out |
| **ConvTranspose** | 2 × C_in × C_out × K_h × K_w × H_out × W_out |
| **MatMul / Gemm** | 2 × M × K × N |
| **BatchNormalization** | 4 × elements（平均、分散、正規化、スケール+シフト） |

MACs（Multiply-Accumulate演算）= FLOPs / 2。これらの推定にはShape Inference（形状推論）を使用して出力次元を決定します。

### メモリ分析

- **重みメモリ**：総パラメータ数 × 4バイト（FP32）。MB単位で表示。
- **ピーク活性化メモリ**：最大の中間テンソル × 3として推定（メモリ内の同時テンソルの概算）。
- **合計メモリ**：重みメモリ + ピーク活性化メモリ。

### ボトルネック診断

100μsを超えるレイヤーは以下のように分類されます：

| カテゴリ | 演算 | 意味 |
|----------|-----------|---------|
| **Compute（計算）** | Conv, MatMul, Gemm, ConvTranspose | CPU/GPUバウンド — 量子化や枝刈りが有効 |
| **Memory（メモリ）** | Reshape, Transpose, Concat, Gather, Slice | メモリバウンド — グラフ最適化や融合が有効 |
| **Other（その他）** | その他すべて | 特殊な最適化が有効な場合がある |

重大度レベル：**High**（>1000μs）、**Medium**（>500μs）、**Low**（>100μs）。

プロファイラは以下も報告します：
- **量子化可能比率**：量子化できる演算の割合
- **推定INT8高速化**：量子化可能比率に基づく概算（完全に量子化可能なモデルで最大約3倍）
- **グラフ深度**：出力から入力までの最長パス（深いグラフはシーケンシャルなボトルネックが多い可能性がある）

**使い方：**
1. **Tools** → **Model Profiler** に移動
2. ONNXモデルを読み込む
3. 実行回数（デフォルト：20）とウォームアップ回数（デフォルト：3）を設定
4. **Profile** をクリック
5. レイテンシ統計（avg、P50、P95、P99）、上位ボトルネックレイヤー、FLOPs内訳、最適化提案を確認

## Model Inspector（モデルインスペクタ）

インスペクタは、モデルの構造と互換性の概要を素早く確認できます。

### グラフメタデータ

- **ファイルサイズ**（MB）
- **Opsetバージョン**：ONNX演算子セットのバージョン。高いバージョンほどより多くの演算をサポート。量子化にはopset ≥ 11が必要。
- **IRバージョン**：ONNX中間表現のバージョン。
- **プロデューサ**：モデルをエクスポートしたフレームワーク（例："pytorch"、"tf2onnx"）。
- **ノード数**：グラフ内の総演算数。
- **演算カウント**：演算タイプ別の内訳（例：Conv: 52、Relu: 48、BatchNormalization: 24）。
- **パラメータ数**：すべてのイニシャライザにわたる総学習可能パラメータ数。

### I/O形状

各入力・出力テンソルについて：
- **名前**：グラフで使用されるテンソル名
- **形状**：次元（`"batch"` や `"height"` などの動的次元を含む場合がある）
- **データ型**：例：`tensor(float)`、`tensor(float16)`、`tensor(int64)`

モデルが入力として何を期待し、何を出力するかを理解するのに役立ちます。

### EP互換性

ssookは、どのExecution Providerがモデルを読み込めるかをテストし、各EPがモデルの演算をどの程度サポートしているかを推定します：

- **利用可能なEP**：システムにインストールされているすべてのEP
- **互換性のあるEP**：モデルを正常に読み込めるEP
- **EP別分析**：互換性のある各EPについて、ssookは以下を推定：
  - **サポートされる演算**：EPでネイティブに実行される演算
  - **フォールバック演算**：CPUにフォールバックする演算
  - **サポート比率**：EPで実行される演算の割合（高いほどGPU活用率が良い）

これは、実際のGPU実行を必要とせずに互換性を推定するために、既知のEP演算サポート（CUDA、TensorRT、OpenVINO、DirectML）のヒューリスティックマッピングを使用します。

**使い方：**
1. **Tools** → **Model Inspector** に移動
2. ONNXモデルを読み込む
3. グラフメタデータ、I/O形状、EP互換性レポートを確認
