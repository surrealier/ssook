## Quantization（量子化）

Quantization（量子化）は、モデルの重み（およびオプションで活性化値）を高精度浮動小数点（FP32）から低精度フォーマット（INT8、FP16）に変換します。これによりモデルサイズが縮小し、特にCPUでの推論が高速化されます。

### Dynamic INT8

**仕組み：**
- 重みはエクスポート時にFP32からINT8に変換されます。
- 活性化値はFP32のまま残り、推論時に**オンザフライ**で量子化されます。
- キャリブレーションデータは不要です。重みの量子化範囲は重みの値から直接計算されます。

**トレードオフ：**
- ✅ 最もシンプルな方法 — モデルを選択して実行するだけ
- ✅ モデルサイズ約50%削減
- ✅ CPUで1.5〜2倍の高速化
- ⚠️ 活性化値の範囲が実行時に推定されるため、Static量子化より最適性が劣る

**ssookでの使い方：**
1. **Tools**タブ → **Calibration / Quantization** に移動
2. ONNXモデルを読み込む
3. **Dynamic INT8** を選択
4. 重みの型（UInt8またはInt8）を選択
5. **Run** をクリック

### Static INT8

**仕組み：**
- 重みと活性化値の両方がINT8に変換されます。
- **キャリブレーションデータセット**（代表的な画像50〜200枚）を使用して、推論中の実際の活性化値の範囲を測定します。
- ssookはキャリブレーション画像をモデルの入力形状に合わせて自動的に前処理（リサイズ、正規化、バッチ化）します。
- QDQ（Quantize-Dequantize）フォーマットは、量子化可能な演算の前後にQ/DQノードを挿入し、最大限の互換性のためにグラフ構造を保持します。

**トレードオフ：**
- ✅ 最高の高速化：CPUで2〜4倍
- ✅ モデルサイズ約75%削減
- ✅ キャリブレーションデータが代表的であれば精度低下は最小限
- ⚠️ ターゲットドメインのキャリブレーション画像が必要

**ssookでの使い方：**
1. **Tools** → **Calibration / Quantization** に移動
2. ONNXモデルを読み込む
3. **Static INT8** を選択
4. キャリブレーション画像ディレクトリと最大画像数を設定
5. オプション（per-channel、QDQフォーマット）を設定
6. **Run** をクリック — プログレスバーでキャリブレーションの進捗が表示されます

### FP16変換

**仕組み：**
- すべてのFP32の重みと定数がFP16（半精度浮動小数点）に変換されます。
- FP16は32ビットの代わりに16ビットを使用し、値あたりのストレージを半減します。
- 表現可能な範囲は小さくなりますが（±65,504 vs ±3.4×10³⁸）、ほとんどのニューラルネットワークの重みには十分です。

**トレードオフ：**
- ✅ モデルサイズ約50%削減
- ✅ ほとんどのモデルで精度低下はほぼなし
- ✅ FP16ハードウェア搭載GPU（NVIDIA Tensor Cores、Apple Neural Engine）で大幅な高速化
- ⚠️ CPUでの高速化は限定的（CPUのFP16サポートは環境による）
- ⚠️ `onnxconverter-common` パッケージが必要

**ssookでの使い方：**
1. **Tools** → **Calibration / Quantization** に移動
2. ONNXモデルを読み込む
3. **FP16** を選択
4. **Run** をクリック

### Mixed Precision（混合精度）

**仕組み：**
- すべてのレイヤーが量子化に対して同じように反応するわけではありません。重みの範囲が広いレイヤーや外れ値が多いレイヤーは、量子化時により多くの精度を失います。
- ssookは量子化可能な各レイヤーの**感度スコア**を計算します：
  ```
  sensitivity = weight_range × (1 + outlier_ratio × 10)
  ```
  ここで `outlier_ratio` は3標準偏差を超える重みの割合です。
- 感度が最も高い上位N%のレイヤーはINT8量子化から**除外**され、FP32のまま保持されます。
- 残りのレイヤーは通常通りINT8に量子化されます。

**トレードオフ：**
- ✅ 完全INT8より高い精度 — 感度の高いレイヤーはFP32のまま
- ✅ 2〜3倍の高速化を達成
- ✅ 自動化 — 手動でのレイヤー選択は不要
- ⚠️ 完全INT8よりわずかに圧縮率が低い

**ssookでの使い方：**
1. **Tools** → **Calibration / Quantization** に移動
2. ONNXモデルを読み込む
3. **Mixed Precision** を選択
4. 除外率を設定（デフォルト：最も感度の高い20%のレイヤー）
5. **Run** をクリック

## Pruning（枝刈り）

Pruning（枝刈り）は、モデルから不要な重みやチャネルを除去し、サイズと計算コストを削減します。

### Weight Pruning（重み枝刈り）

**仕組み：**
- **マグニチュードベースの非構造化枝刈り**：絶対値が最も小さい重みは出力への寄与が最も少なく、ゼロに設定できます。
- `sparsity_ratio`（例：0.3）が与えられると、マグニチュードで下位30%の重みがゼロ化されます。
- Conv、MatMul、Gemm、ConvTranspose演算の重みテンソルが対象です。
- 結果のスパースモデルは圧縮時により小さくなり、スパース計算をサポートするランタイムではより高速になります。

**トレードオフ：**
- ✅ 10〜30%のサイズ削減（圧縮でさらに削減）
- ✅ 適度なスパース性（≤30%）では精度低下は最小限
- ⚠️ 高速化はランタイムのスパースサポートに依存 — 標準のONNX Runtimeはスパース演算を高速化しない
- ⚠️ 高いスパース性（>50%）は精度を著しく低下させる可能性がある

**ssookでの使い方：**
1. **Tools** → **Model Optimization** に移動
2. ONNXモデルを読み込む
3. **Weight Pruning** を選択
4. スパース率を設定（0.0〜1.0、推奨：0.2〜0.3）
5. **Run** をクリック

### Channel Pruning（チャネル枝刈り）

**仕組み：**
- **L1ノルムベースの構造化枝刈り**：各Convレイヤーについて、各出力チャネルの重みのL1ノルム（絶対値の合計）を計算します。
- L1ノルムが最も低いチャネルは、そのレイヤーの出力への寄与が最も少ないです。
- L1ノルムで下位N%のチャネルが完全に除去されます — 重みと対応するバイアス値の両方が除去されます。
- Weight Pruningとは異なり、これは**構造化**されています：結果のモデルはチャネル数が少なくなり、特別なスパースサポートなしにあらゆるハードウェアで高速に動作します。

**トレードオフ：**
- ✅ 枝刈りされたチャネルに比例した直接的なFLOPs削減
- ✅ すべてのハードウェアで高速化（スパースランタイム不要）
- ⚠️ 精度回復のためにファインチューニングが必要な場合がある
- ⚠️ レイヤーがゼロにならないよう最小チャネル数が適用される

**ssookでの使い方：**
1. **Tools** → **Model Optimization** に移動
2. ONNXモデルを読み込む
3. **Channel Pruning** を選択
4. 枝刈り率（推奨：0.1〜0.3）と最小チャネル数（デフォルト：4）を設定
5. **Run** をクリック

## Graph Optimization（グラフ最適化）

Graph Optimization（グラフ最適化）は、モデルの数学的な動作を変えることなく、計算グラフを再構成して冗長性を排除し、演算を融合します。

### ORT Graph Optimizer

**仕組み：**
- ONNX Runtimeの組み込みグラフ最適化パスを使用します。
- **Constant folding（定数畳み込み）**：入力がすべて定数である演算を事前に計算します（例：Shape → Gather → Unsqueezeチェーン）。
- **Operator fusion（演算子融合）**：隣接する演算を単一の最適化カーネルに統合します：
  - Conv + BatchNormalization → 単一のConv（BNパラメータがConvの重みに畳み込まれる）
  - Conv + Relu → ConvRelu融合カーネル
  - MatMul + Add → 融合Gemm
  - Conv + Add + Relu → 単一の融合カーネル
- 3つの最適化レベル：**Basic**（定数畳み込み）、**Extended**（+ 共通部分式除去）、**All**（+ 高度な融合）。

**トレードオフ：**
- ✅ 5〜15%のレイテンシ削減
- ✅ 精度低下ゼロ — 数学的に等価
- ✅ 常に安全に適用可能

### ONNX Simplifier

**仕組み：**
- `onnxsim` ライブラリを使用して、ONNXグラフレベルでShape Inference（形状推論）とConstant Folding（定数畳み込み）を実行します。
- 可能な場合は動的形状を解決し、冗長な演算を簡略化して、よりクリーンなグラフを生成します。

**トレードオフ：**
- ✅ よりクリーンなグラフ、多くの場合ファイルサイズも縮小
- ✅ 精度低下ゼロ
- ⚠️ `onnxsim` パッケージが必要

### Dead Node Elimination（不要ノード除去）

**仕組み：**
- 出力ノードからグラフを逆方向にたどり、到達可能なすべてのノードを見つけます。
- 出力から到達できないノードはすべて除去されます。
- モデルのエクスポートや以前の変換で残ったノードをクリーンアップします。

**トレードオフ：**
- ✅ グラフサイズの削減
- ✅ 精度低下ゼロ
- ✅ 他の最適化の前に実行すると効果的

**ssookでのグラフ最適化の使い方：**
1. **Tools** → **Model Optimization** に移動
2. ONNXモデルを読み込む
3. 最適化手法を選択
4. ORT Graph Optimizerの場合、レベル（Basic / Extended / All）を選択
5. **Run** をクリック

## 最適化パイプライン

ssookでは、複数の最適化を**連鎖的に**実行できます。例：

1. Dead Node Elimination → グラフのクリーンアップ
2. ORT Graph Optimizer → 演算の融合
3. Channel Pruning → 重要でないチャネルの除去
4. Dynamic INT8 Quantization → 重みの圧縮

各ステップは前のステップの出力を入力として使用します。パイプラインは各段階でのサイズ変化と結果を追跡します。

**使い方：**
1. **Tools** → **Model Optimization** に移動
2. **Model Diagnosis** 機能を使用して推奨される最適化ステップを取得
3. 診断エンジンがモデルの特性に基づいて優先順位付きパイプラインを提案
4. 推奨パイプラインを適用するか、ステップをカスタマイズ

## 性能比較

CPU（Intel i7-1165G7）での一般的な結果：

| モデル | 手法 | サイズ | 推論時間 | mAP@50 |
|-------|--------|------|---------------|--------|
| YOLOv8n FP32 | オリジナル | 12.2 MB | 約45 ms | 37.3 |
| YOLOv8n INT8 (Dynamic) | Dynamic Quantization | 3.4 MB | 約25 ms | 36.8 |
| YOLOv8n INT8 (Static) | Static Quantization | 3.2 MB | 約18 ms | 37.0 |
| YOLOv8n FP16 | FP16変換 | 6.1 MB | 約38 ms | 37.3 |

> 結果はハードウェアとモデルによって異なります。最適化後は必ずssookのEvaluationタブで精度を検証してください。

## Tips＆注意事項

1. **まずGraph Optimizationから始める** — 精度低下がなく（無料）、後続の最適化をより効果的にします。
2. **最初にModel Diagnosisを使用する** — モデルを分析し、最適な最適化戦略を推奨します。
3. **量子化後に精度を検証する** — Evaluationタブを使用して最適化前後のmAPを比較してください。
4. **キャリブレーションデータが重要** — Static INT8では、実際のデプロイドメインの画像を使用してください（通常50〜200枚で十分）。
5. **opsetバージョンを確認する** — 量子化にはopset ≥ 11が必要です。低いopsetのモデルは失敗する可能性があります。
6. **CenterNet / DETRモデル** — 一部のカスタム演算はINT8をサポートしない場合があります。代わりにFP16またはMixed Precisionを使用してください。
7. **Channel Pruningにはファインチューニングが必要** — 最良の結果を得るには、枝刈り後にトレーニングデータでモデルをファインチューニングしてください。
